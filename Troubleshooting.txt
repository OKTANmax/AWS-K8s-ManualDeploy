Challenges & Troubleshooting
1. Remote Access to Isolated Instances
Issue: Unable to establish SSH connectivity to EC2 instances due to their placement within Private Subnets (as per security hardening best practices).

Resolution: Instead of using a Bastion Host or opening SSH ports, I implemented AWS Systems Manager (SSM) Session Manager. This was achieved by attaching the necessary IAM policies to the instance profiles, allowing secure, browser-based shell access without exposing public IP addresses or managing SSH keys.

2. SSM Agent Connectivity Failures
Issue: Despite correct IAM roles and Security Group configurations, the SSM agent remained unreachable, preventing remote management.

Analysis: Investigation of the VPC Route Tables revealed a configuration drift; the default route (0.0.0.0/0) was pointing to a deprecated/deleted NAT Gateway.

Resolution: Re-configured the Route Table to point to the active NAT Gateway, restoring outbound internet access for the instances to register with the SSM service endpoints.

3. Load Balancer Health Check Flapping & Inter-Node Communication
Issue: The Application Load Balancer (ALB) reported targets as "Unhealthy" (flapping), and internal cluster traffic was inconsistent despite specific ports (6443, 10250, etc.) being open.

Analysis: The overly restrictive "Least Privilege" rules in the Security Groups were inadvertently blocking essential Kubelet-to-API and Pod-to-Pod control plane traffic required by the CNI.

Resolution: Refined the security architecture to allow all traffic between the Control Plane and Worker Node Security Groups. By using Security Group Referencing (rather than CIDR blocks), I maintained a hardened posture while ensuring seamless, unrestricted communication across the cluster's internal fabric.
